# Python Code Review Skill

专业的 Python 代码评审指南，帮助识别代码中的问题并提供改进建议。

## 评审流程

### 1. 首先阅读代码
使用 Read 工具完整阅读目标文件，理解代码的整体结构和意图。

### 2. 按类别检查

评审时按以下优先级顺序检查：

#### 高优先级 - 安全问题
- **SQL 注入**: 检查是否使用字符串拼接构建 SQL 语句
- **命令注入**: 检查 `subprocess`/`os.system` 是否接收用户输入
- **路径遍历**: 检查文件路径操作是否验证用户输入
- **敏感信息泄露**: 检查是否硬编码密码、API Key、密钥
- **XSS/CSRF**: Web 应用中检查输入输出是否正确处理

#### 高优先级 - 潜在 Bug
- **空值处理**: 检查可能的 `None`/空列表访问
- **类型错误**: 检查类型假设是否正确
- **边界条件**: 检查循环边界、索引越界风险
- **异常处理**: 检查是否捕获过于宽泛的异常 (`except:` / `except Exception`)
- **资源泄漏**: 检查文件、连接是否正确关闭（优先建议使用 `with` 语句）

#### 中优先级 - 性能问题
- **循环内重复计算**: 检查是否有可移出循环的计算
- **低效数据结构**: 检查是否在列表中进行频繁查找（建议用 dict/set）
- **N+1 查询**: 数据库相关代码检查
- **内存效率**: 检查大列表/字典的创建和复制

#### 中优先级 - 代码质量
- **命名规范**: 检查变量/函数命名是否清晰（snake_case）
- **代码重复**: 检查相似代码是否可以抽取
- **函数长度**: 单个函数是否过长（建议 < 50 行）
- **参数过多**: 函数参数是否超过 5 个
- **魔法数字**: 检查是否有未解释的硬编码数值

#### 低优先级 - 最佳实践
- **类型注解**: 是否添加了类型提示
- **文档字符串**: 关键函数是否有 docstring
- **导入顺序**: 是否按标准分组（stdlib → third-party → local）
- **代码风格**: 是否符合 PEP 8 规范

## 评审原则

### 保持客观
- 只报告实际观察到的问题，不猜测意图
- 基于代码实际内容评审，不凭空假设

### 提供具体建议
- 每个问题都要有具体的改进建议
- 建议应该可操作，而非泛泛而谈
- 如果可能，提供代码示例

### 区分严重程度
- 🔴 高优先级：可能导致运行时错误、安全漏洞、数据丢失
- 🟡 中优先级：影响性能、可维护性，但不会导致严重问题
- 🟢 低优先级：风格问题、最佳实践建议

### 不要过度评审
- 不建议添加不必要的抽象或过度工程化
- 简单的代码不需要复杂的改进
- 如果代码已经很好，明确说明

## 注意事项

- **不要修改代码**：评审只输出报告，不直接修改文件（除非用户明确要求）
- **关注评审范围**：只评审用户指定的文件或代码段
- **保持简洁**：报告应简洁明了，避免冗长解释
- **理解上下文**：考虑代码的业务场景，不做脱离实际的评判

## 常见问题检查清单

```markdown
- [ ] SQL 查询是否使用参数化？
- [ ] 用户输入是否经过验证？
- [ ] 文件操作是否使用 with 语句？
- [ ] 异常处理是否具体？
- [ ] 是否有未处理的 None 情况？
- [ ] 循环是否有优化空间？
- [ ] 函数是否有清晰的命名？
- [ ] 是否有重复代码？
```
